# Conversion plan: `python-agentframework-demos` → `dotnet-agentframework-demos`

This plan converts the Python repo at `C:\g\python-agentframework-demos` into a .NET version at `C:\g\dotnet-agentframework-demos`. The goal is full feature parity with a .NET flavor — same examples, same infra, same dev container services, but built on the .NET edition of the Microsoft Agent Framework.

We will execute this in ordered, isolated stages. Each stage is a self-contained chunk of work that can be reviewed before moving on.

## Target .NET stack (mapping)

The .NET version uses **.NET 10 file-based apps** (`dotnet run app.cs`). Each example is a single `.cs` file that declares its NuGet dependencies inline via `#:package` directives — no `.csproj`, no `.sln`, no `Directory.*.props`. This is the closest match to Python's "one runnable script per example" model.

| Concern | Python | .NET 10 equivalent |
|---|---|---|
| Runtime | Python 3.12 + `uv` | .NET 10 SDK |
| Project model | one `.py` per example | one `.cs` per example, run with `dotnet run examples/agent_basic.cs` |
| Dependency mgmt | `pyproject.toml` / `uv.lock` | inline `#:package Foo@x.y.z` at top of each `.cs` file |
| Agent framework | `agent-framework-core`, `agent-framework-openai`, `-orchestrations` | `Microsoft.Agents.AI`, `Microsoft.Agents.AI.OpenAI`, `Microsoft.Agents.AI.Workflows` |
| Auth | `azure-identity` (DefaultAzureCredential async) | `Azure.Identity` (`DefaultAzureCredential`) |
| OpenAI SDK | `openai` python | `OpenAI` (official .NET) + `Azure.AI.OpenAI` |
| Console UI | `rich` | `Spectre.Console` |
| Schema/validation | `pydantic` | `System.Text.Json` + records |
| .env loading | `python-dotenv` / `dotenv-azd` | `DotNetEnv` |
| OTel | `opentelemetry-exporter-otlp-proto-grpc`, `azure-monitor-opentelemetry` | `OpenTelemetry.Exporter.OpenTelemetryProtocol`, `Azure.Monitor.OpenTelemetry.Exporter` |
| PostgreSQL + pgvector | `psycopg`, `pgvector` | `Npgsql`, `Pgvector.Npgsql` |
| Redis | `redis-py` (via MAF Redis pkg) | `StackExchange.Redis`, `Microsoft.Agents.AI.Redis` (if available) |
| MCP | `fastmcp` (server) + MAF MCP client | `ModelContextProtocol` (csharp-sdk) for both client + server |
| PDF / Markdown | `markitdown[pdf]` | `UglyToad.PdfPig` + `Markdig` (or shell out to `markitdown` CLI) |
| Evaluation / Red-team | `azure-ai-evaluation[redteam]` | `Azure.AI.Evaluation` (preview) — feature parity verified per-example |
| Mem0 | `agent-framework-mem0` | Mem0 .NET SDK if available, else interop with REST |

If a .NET package does not yet exist with feature parity, the corresponding example will be flagged and either (a) reimplemented against the lower-level primitive (OpenAI SDK, raw HTTP), or (b) deferred with a clear `[NEEDS NUGET]` note.

## Stage 1 — Dev container

Goal: produce `.devcontainer/{devcontainer.json,docker-compose.yml,Dockerfile}` that gives a working .NET dev environment with the same supporting services (Postgres+pgvector, Redis, Aspire Dashboard).

### Changes from Python version

**`Dockerfile`**
- Base image: `mcr.microsoft.com/devcontainers/dotnet:1-10.0-bookworm`
- Drop: `uv`, Rust toolchain, `git-lfs`, `pyproject.toml`/`uv sync` warm-up — none needed for .NET 10
- No `dotnet restore` warm-up (file-based apps restore lazily on first `dotnet run`)

**`docker-compose.yml`**
- `app` service: builds from new Dockerfile, mounts `..:/workspace`
- Env vars on `app`: same OTEL/REDIS/POSTGRES URLs
- `db`, `redis`, `aspire-dashboard`: **unchanged** (same images, same ports)

**`devcontainer.json`**
- `name`: `dotnet-agentframework-demos`
- `forwardPorts`: same (`5432, 6379, 18888`)
- `features`: keep `azd`, `azure-cli`, `github-cli`
- `customizations.vscode.extensions`: replace Python extensions with:
  - `ms-dotnettools.csharp`
  - `ms-dotnettools.csdevkit`
  - `ms-azuretools.vscode-bicep`
  - `ms-ossdata.vscode-pgsql`
- `customizations.vscode.settings`: drop `python.*`; no `dotnet.defaultSolution` (no .sln exists)
- `postCreateCommand`: `dotnet --info` (just a sanity check; nothing to restore up front)
- `remoteUser`: `vscode`

**Acceptance**: container builds; `dotnet --info` works inside; `nc -z db 5432`, `nc -z redis 6379`, `curl http://aspire-dashboard:18888` reachable from `app`.

## Risks & open questions

1. **NuGet version pinning**: the Python repo pins MAF `1.0.0` / `1.0.0b260402`. The .NET edition versions may differ; the canonical `#:package` block will be resolved to the latest stable + matching previews when Stage 4 lands, then re-used across all example files.
2. **Mem0 .NET**: no first-party MAF Mem0 package known. `agent_memory_mem0` may need a thin Mem0 REST client.
3. **`azure-ai-evaluation[redteam]`**: .NET equivalent (`Azure.AI.Evaluation`) is in preview and may not cover IntentResolution/ToolCallAccuracy yet. The 4 evaluation examples will be checked per-feature during Stage 5.
4. **`markitdown`** has no native .NET equivalent. Options: `UglyToad.PdfPig` for PDFs + `Markdig`, or shell out to the `markitdown` Python CLI from inside the dev container.
5. **`agent-framework-devui`** is a web UI for MAF Python. The .NET equivalent (if any) will be identified before Stage 5; otherwise that example is dropped or replaced.

These risks affect Stage 5 only — they do not block Stages 1–3.
